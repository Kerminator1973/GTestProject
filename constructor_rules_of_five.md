# Правила трёх и правила пяти

Правило трёх (также известное как "Закон Большой Тройки") действовало до появления C++11. Оно гласило, что если класс или структура определяет один из следующих методов, то необходимо явно определить все три:

- Конструктор по умолчанию
- Конструктор копирования — позволяет создавать новый объект как копию существующего
- Оператор присваивания копированием — позволяет одному уже существующему объекту принять состояние другого существующего объект

Эти методы являются особыми членами-функциями. Если один из них требует явного определения, это означает, что версия, сгенерированная компилятором, не удовлетворяет потребностям класса в одном случае и, вероятно, не подойдёт в остальных. 

**Почему это важно?** Неявно определённые конструктор копирования и оператор присваивания выполняют поверхностное копирование (shallow copy), то есть просто копируют все члены-данные класса. Это может привести к проблемам, если класс управляет внешними ресурсами (например, выделяет память, работает с файловыми дескрипторами или сетевыми соединениями). В таком случае два объекта будут «думать», что они владеют одним и тем же ресурсом, и попытаются его освободить при уничтожении, что вызовет ошибки. 

Поправка к правилу трёх — если используется идиома **RAII** (Resource Acquisition Is Initialization), то деструктор может остаться неопределённым.

>Если мы определяем деструктор, то автоматическая генерация остальных конструкторов и операторов присваивания не будет работать.

## Правило пяти

С появлением C++11 и введением семантики перемещения правило трёх расширилось до правила пяти. Теперь, если явно определяется один из следующих пяти специальных методов класса, вероятно, нужно явно определить и остальные четыре:

- Конструктор по умолчанию
- Конструктор копирования
- Оператор присваивания копированием
- Конструктор перемещения — позволяет "забирать" ресурс у временного объекта без копирования, что повышает эффективность
- Оператор присваивания перемещением — аналогично конструктору перемещения, но для операции присваивания

Семантика перемещения позволяет оптимизировать работу с ресурсами, особенно в контейнерах (например, std::vector), где при реаллокации часто происходит копирование объектов. Использование noexcept для перемещающих методов важно, чтобы контейнеры могли выбирать перемещение вместо копирования.

Пример реализации:

```cpp
#include <cstring>

class RFive {
private:
    char* cstring;

public:
    // Конструктор
    RFive(const char* arg) : cstring(new char[std::strlen(arg) + 1]) {
        std::strcpy(cstring, arg);
    }

    // Деструктор
    ~RFive() {
        delete cstring;
    }

    // Конструктор копирования
    RFive(const RFive& other) {
        cstring = new char[std::strlen(other.cstring) + 1];
        std::strcpy(cstring, other.cstring);
    }

    // Конструктор перемещения
    RFive(RFive&& other) noexcept {
        cstring = other.cstring;
        other.cstring = nullptr;
    }

    // Оператор присваивания копированием
    RFive& operator=(const RFive& other) {
        if (this == &other) return *this;
        char* tmp_cstring = new char[std::strlen(other.cstring) + 1];
        std::strcpy(tmp_cstring, other.cstring);
        delete cstring;
        cstring = tmp_cstring;
        return *this;
    }

    // Оператор присваивания перемещением
    RFive& operator=(RFive&& other) noexcept {
        if (this == &other) return *this;
        delete cstring;
        cstring = other.cstring;
        other.cstring = nullptr;
        return *this;
    }
};
```

## Дополнительные рекомендации

Идиома копирования и обмена (copy-and-swap) позволяет избежать дублирования кода, используя повторно код конструктора копирования. При такой реализации отпадает нужда в проверке на самоприсваивание.

Пример:

```cpp
Array &operator=(Array other) noexcept {
    swap(*this, other);
    return *this;
}
```

В современном C++ рекомендуется по возможности использовать правило нуля — не определять эти методы явно, а делегировать управление ресурсами RAII-типам (например, std::unique_ptr, std::shared_ptr, std::vector). В этом случае компилятор сгенерирует корректные операции сам.
