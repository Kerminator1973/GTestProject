# CRTP (Curiously Recurring Template Pattern)

Сразу следует заметить, что шаблон проектирования относится к устаревающим и начиная с C++ 23 заменяется другим подходом.

Важно сказать о том, что CRTP, который реализует статический полиморфизм, не заменяет традиционный динамический полиморфизм, а предоставляет альтернативный способ полиморфного поведения с меньшим расходом вычислительных ресурсов, но с увеличением объёма кода и потерей динамического поведения.

## Что такое CRTP в C++

**CRTP** (Curiously Recurring Template Pattern, "странно рекурсивный шаблон проектирования") — идиома в языке C++, при которой класс наследуется от шаблонного класса, используя себя в качестве параметра шаблона.

Структура выглядит так: класс `X` наследуется от класса‑шаблона `Y<X>`.

1. **Базовый класс** определяется как шаблон, который принимает тип производного класса в качестве параметра. Это позволяет базовому классу обращаться к членам и методам производного класса
2. **Производный класс** наследуется от базового класса‑шаблона, передавая себя в качестве аргумента шаблона. Это создаёт рекурсивную зависимость
3. В базовом классе можно использовать `static_cast` для приведения указателя `this` к типу производного класса — и вызывать его методы напрямую

## Синтаксис

```cpp
template <typename T>
class Base {
    // методы Base могут использовать T для доступа к членам Derived
};

class Derived : public Base<Derived> {
    // ...
};
```

Пример из книги "Software Architecture with C++" by Adrian Ostrowski:

```cpp
template <typename ConcreteItem> class GlamorousItem {
public:
    void appear_in_full_glory() {
        static_cast<ConcreteItem *>(this)->appera_in_full_glory();
    }
}
```

Производный класс:

```cpp
class PinkHeels : public GlamorousItem<PinkHeels> {
public:
    void appear_in_full_glory() {
        std::cout << "Pink high heels suddenly appeared in all their beauty\n";
    }
}

class GoldenWatch : public GlamorousItem<GoldenWatch> {
public:
    void appear_in_full_glory() {
        std::cout << "Everyone wanted to watch this watch\n";
    }
}
```

## Основные цели использования CRTP

* **Статический полиморфизм**. Позволяет достичь полиморфизма на этапе компиляции — без накладных расходов на динамический полиморфизм (виртуальные функции и таблицы виртуальных методов).
* **Избегание дублирования кода**. Общая функциональность реализуется в базовом классе‑шаблоне и автоматически доступна всем производным классам.
* **Расширение интерфейса**. Базовый класс может добавлять методы в интерфейс производного класса.
* **Оптимизация производительности**. Поскольку диспетчеризация вызовов происходит на этапе компиляции, код выполняется быстрее, чем при использовании виртуальных функций.

## Примеры использования

**Пример 1. Счётчик объектов**

Базовый класс `Counter` отслеживает количество созданных и существующих объектов для каждого производного класса:

```cpp
template<typename T>
class Counter {
public:
    Counter() { ++created; ++alive; }
    Counter(const Counter&) { ++created; ++alive; }
    ~Counter() { --alive; }

    static int howManyAlive() { return alive; }
    static int howManyCreated() { return created; }

private:
    static int created;
    static int alive;
};

template<typename T> int Counter<T>::created(0);
template<typename T> int Counter<T>::alive(0);

class MyClass : public Counter<MyClass> { /* ... */ };
class AnotherClass : public Counter<AnotherClass> { /* ... */ };
```

**Пример 2. Статический полиморфизм**

Базовый класс `Vehicle` предоставляет единый интерфейс для получения количества колёс, делегируя реализацию производным классам:

```cpp
template<typename Derived>
class Vehicle {
public:
    int getNumberOfWheels() const {
        return static_cast<const Derived*>(this)->getNumberOfWheelsImpl();
    }
};

class Car : public Vehicle<Car> {
public:
    int getNumberOfWheelsImpl() const { return 4; }
};

class Bicycle : public Vehicle<Bicycle> {
public:
    int getNumberOfWheelsImpl() const { return 2; }
};
```

## Плюсы CRTP

* Высокая производительность (статическая диспетчеризация)
* Отсутствие накладных расходов виртуальных вызовов
* Возможность повторного использования кода
* Типобезопасность на этапе компиляции

## Минусы CRTP

* Усложнение кода и снижение читаемости
* Риск неопределённого поведения при неправильном наследовании (например, если производный класс наследуется от `Base<AnotherClass>` вместо `Base<Derived>`).
* Возможность скрытия методов базового класса при переопределении в производном классе
* Увеличение размера скомпилированного кода из‑за генерации отдельных экземпляров шаблонов для каждого производного класса

## Когда стоит использовать CRTP

* В библиотеках и фреймворках, где важна производительность
* При необходимости добавить общую функциональность нескольким несвязанным классам
* Когда динамический полиморфизм не нужен или нежелателен из‑за накладных расходов
* В сценариях, где требуется статическая диспетчеризация и типобезопасность

## С++ 23

В C++23 CRTP не был полностью заменён на другой механизм, но его использование во многих случаях стало менее актуальным благодаря появлению явного объектного параметра (_explicit object parameter_), также известного как _deducing this_. Эта новая возможность языка позволяет упростить код и избежать некоторых сложностей, связанных с CRTP.

Deducing this позволяет явно указывать объект, на который вызывается метод, в списке параметров нестатической функции-члена. При этом тип объекта, его квалификаторы (const, volatile) и категория значения (lvalue/rvalue) выводятся автоматически через шаблоны. Синтаксис выглядит так:

```cpp
struct X {
    template<typename Self>
    void foo(this Self&& self) {
        // self — явный параметр, представляющий объект
    }
};
```

CRTP часто использовался для реализации статического полиморфизма и миксинов, когда базовый класс шаблонизировался на производный, чтобы получить доступ к его методам через static_cast. С deducing this необходимость в таком подходе отпадает: вместо приведения this к производному типу можно напрямую использовать параметр self, чей тип автоматически выводится как правильный производный тип (с учётом квалификаторов).

До CRTP:

```cpp
template <typename Derived>
class Base {
public:
    void do_something() {
        static_cast<Derived*>(this)->impl();
    }
};

class Derived : public Base<Derived> {
public:
    void impl() {
        std::cout << "Impl in Derived\n";
    }
};
```

C++23:

```cpp
class Base {
public:
    template <typename Self>
    void do_something(this Self& self) {
        self.impl();
    }
};

class Derived : public Base {
public:
    void impl() {
        std::cout << "Impl in Derived\n";
    }
};
```

В приведённом выше коде нет необходимости в шаблонном базовом классе и приведении типов — доступ к методу impl() получается напрямую через параметр self.
