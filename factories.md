# Использование фабричных методов и классов

**Фабричные методы**, также называемые _names constructor idiom_, состоит в том, статический член класса вызывает приватный конструктор, чтобы создать экземпляр класса. Например:

```cpp
class Pixel {
    public:
        static Pixel fromRgba(char r, char g, char b, char a) {
            return Pixel {r, g, b, a};
        }
        static Pixel fromBgra(char b, char g, char r, char a) {
            return Pixel {r, g, b, a};
        }
    // ...
    private:
        Pixel(char r, char g, char b, char a) : r_(r), g_(g), b_(b), a_(a) {}
        char r_, g_, b_, a_;
}
```

Для чего это нужно:

- Когда существует слишком много разных способов создать объект, что может приводить к частым ошибкам. В приведённом выше примере было бы проблемным создать два конструктора для Rgba и Bgra
- Когда необходимо создавать объект на heap-е, или в какой-то конкретной области памяти
- Когда при создании объекта может возникнуть исключение, но вы не можете сбрасывать исключение и его нужно подавить

Ключевые черты:

- Реализация через наследование. Базовый класс объявляет фабричный метод (часто виртуальный или абстрактный), а подклассы переопределяют его, чтобы возвращать объекты нужного типа.
- Создаёт один объект. Фокус на порождении одного продукта (или объектов одной иерархии)
- Проще в реализации. Меньше кода и классов по сравнению с абстрактной фабрикой
- Гибкость. Легко добавить новый тип продукта — достаточно создать новый подкласс и переопределить фабричный метод

## Фабричные классы

**Фабричный класс** / **Абстрактная фабрика** (_Abstract Factory_): Предоставляет интерфейс для создания семейств взаимосвязанных объектов без указания их конкретных классов.

Ключевые черты:

- Работа с семействами объектов. Создаёт не один объект, а группу связанных объектов (например, кнопки + меню + окна для одной ОС)
- Интерфейс фабрики. Определяет несколько фабричных методов, каждый из которых отвечает за создание одного типа объекта из семейства
- Конкретные фабрики. Реализуют интерфейс абстрактной фабрики и возвращают конкретные продукты для определённого семейства
- Более сложная структура, чем **фабричный метод**. Требует больше классов и кода
- Согласованность семейства. Гарантирует, что все созданные объекты принадлежат одному семейству (например, все элементы интерфейса в одном стиле)

Пример использования фабрики:

```cpp
class Application {
private:
    std::unique_ptr<GUIFactory> factory;
public:
    Application(std::unique_ptr<GUIFactory> f) : factory(std::move(f)) {}

    void createUI() {
        auto button = factory->createButton();
        auto menu = factory->createMenu();
        button->render();
        menu->render();
    }
};
```
